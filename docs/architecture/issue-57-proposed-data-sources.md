# Architecture: Agent-Proposed Data Sources (Issue #57)

## Approach

Introduce a `ProposedDataSource` CRD that agents can create to nominate new data
sources for the enterprise catalog. The CRD is namespace-scoped so each agent session
sees only its own proposals. `requestedBy` is always set server-side from the resolved
session namespace — never from agent input. Operator approval is required before a
proposal is promoted to a `DataSource`. Two new MCP tools (`propose_data_source`,
`get_data_source_proposal`) follow the same poll-until-settled pattern as
`service_status`.

## Data / API Changes

### `ProposedDataSource` CRD

```go
// api/v1alpha1/proposeddatasource_types.go

// +kubebuilder:object:root=true
// +kubebuilder:resource:scope=Namespaced
// +kubebuilder:subresource:status
// +kubebuilder:printcolumn:name="Kind",type=string,JSONPath=`.spec.kind`
// +kubebuilder:printcolumn:name="Phase",type=string,JSONPath=`.status.phase`
// +kubebuilder:printcolumn:name="Age",type=date,JSONPath=`.metadata.creationTimestamp`

// allowedKinds is the validated allowlist for ProposedDataSource.spec.kind.
var allowedKinds = map[string]bool{
    "postgres": true, "mysql": true, "s3": true, "http-api": true, "kafka": true,
}

type ProposedDataSourceSpec struct {
    // Kind is the type of data source. Must be one of: postgres, mysql, s3, http-api, kafka.
    // +kubebuilder:validation:Enum=postgres;mysql;s3;http-api;kafka
    Kind string `json:"kind"`

    // Description is a human-readable description.
    Description string `json:"description"`

    // Rationale explains why this data source is needed.
    Rationale string `json:"rationale"`

    // RequestedBy is set by the platform from the agent's session namespace.
    // Agents cannot supply this field — it is always overwritten server-side.
    // +optional
    RequestedBy string `json:"requestedBy,omitempty"`

    // Schema is an optional description of the data schema (tables, API paths, etc.).
    // +optional
    Schema string `json:"schema,omitempty"`

    // Tags are optional labels for filtering.
    // +optional
    Tags []string `json:"tags,omitempty"`

    // AppName is an optional IAF Application in the same namespace whose URL will
    // be populated in status once the app is Running. Must pass ValidateAppName.
    // +optional
    AppName string `json:"appName,omitempty"`
}

type ProposedDataSourceStatus struct {
    // Phase is Pending, Approved, or Rejected.
    // +kubebuilder:validation:Enum=Pending;Approved;Rejected
    Phase string `json:"phase,omitempty"`

    // RejectionReason is set when Phase is Rejected.
    // +optional
    RejectionReason string `json:"rejectionReason,omitempty"`

    // PromotedDataSourceName is the name of the created DataSource when Phase is Approved.
    // +optional
    PromotedDataSourceName string `json:"promotedDataSourceName,omitempty"`

    // AppURL is the resolved URL of the referenced IAF Application when it is Running.
    // Populated by the platform controller to help operators fill in DataSource credentials.
    // +optional
    AppURL string `json:"appURL,omitempty"`
}
```

No `secretRef` field anywhere in `ProposedDataSourceSpec` — credentials are always
supplied by the operator when they promote the proposal to a `DataSource`.

### `propose_data_source` tool

```
Input:
  session_id  string  (required)
  kind        string  (required; validated against allowlist)
  description string  (required)
  rationale   string  (required)
  schema      string  (optional)
  tags        string  (optional; comma-separated)
  app_name    string  (optional; must exist in session namespace)

Output (on success):
  {
    "proposalName": "proposed-ds-<generated-suffix>",
    "phase": "Pending",
    "message": "Proposal submitted. Poll get_data_source_proposal until phase is Approved or Rejected."
  }
```

Validation order in the tool handler:
1. `deps.ResolveNamespace(input.SessionID)` → `namespace`
2. Validate `kind` against `allowedKinds` (return error with allowlist on failure)
3. If `app_name` non-empty: call `ValidateAppName`; verify Application exists in
   `namespace` (return error if not found)
4. Check idempotency: list `ProposedDataSource` in `namespace` where
   `spec.kind == input.Kind && spec.appName == input.AppName`; if exact match found,
   return existing proposal name rather than creating a duplicate
5. Build CR with `requestedBy = namespace` (overrides any value if somehow present in
   input, though the input struct has no such field)
6. Create CR in `namespace`

Proposal name: `iaf-proposal-<random-5-char-suffix>` (prefixed to distinguish from
DataSource names). Use `names.SimpleNameGenerator` from `k8s.io/apimachinery` or
generate with `fmt.Sprintf("iaf-proposal-%s", randSuffix(5))`.

### `get_data_source_proposal` tool

```
Input:
  session_id     string  (required)
  proposal_name  string  (required)

Output:
  {
    "proposalName": "iaf-proposal-xxxxx",
    "phase": "Pending | Approved | Rejected",
    "rejectionReason": "..." (if Rejected),
    "promotedDataSourceName": "..." (if Approved),
    "appURL": "..."  (if app is Running)
  }
```

The tool gets the `ProposedDataSource` from the session namespace. If not found, returns
a clear error with the proposal name. This prevents cross-namespace reads: agents can
only read proposals in their own namespace.

### `iaf://catalog/data-sources` resource update

Add a `pendingProposals` count to the JSON output:
```json
{
  "dataSources": [...],
  "pendingProposals": 3
}
```
The count is fetched by listing `ProposedDataSource` across all namespaces (cluster-wide
read, metadata only) and filtering for `status.phase == "Pending"` or empty phase.
**No proposal content (description, rationale) is returned** — only the count. This
gives the operator a signal without leaking proposal details to other agents reading the
catalog.

## Multi-tenancy & Shared Resource Impact

- `ProposedDataSource` is namespace-scoped: agents cannot read proposals from other
  namespaces.
- `requestedBy` is set server-side from `ResolveNamespace` — prevents impersonation.
- `list_data_sources` (existing tool) only lists approved `DataSource` objects (cluster-
  scoped, operator-created). ProposedDataSources do not appear there.
- The data catalog resource (above) adds only a total count — no cross-namespace content
  is exposed to individual agents.
- Operators (cluster-admin) can read proposals across namespaces via `kubectl get
  proposeddatasources --all-namespaces` to perform the approval workflow.

## Security Considerations

- **No `secretRef` in spec**: this is the primary security control. An agent can never
  inject credentials through a proposal. Credentials are always operator-supplied at
  promotion time.
- **`requestedBy` immutability**: after creation, the platform must not allow agents to
  patch `requestedBy`. The RBAC rule for agents grants only `create`, `get`, `list`,
  `watch` — no `update` or `patch` on `proposeddatasources`. The status subresource
  update (operator approval) requires a separate RBAC rule.
- **Kind allowlist**: enforced both in the tool (application layer) and via
  `+kubebuilder:validation:Enum` in the CRD (apiserver layer). Defense in depth.
- **AppName validation**: before recording `appName`, verify the Application exists in
  the session namespace. This prevents agents from referencing apps in other namespaces.
- **Schema field**: the schema string is operator-advisory only. No code evaluates or
  executes it. Still, validate that it doesn't exceed a reasonable size limit (e.g. 10 KB)
  to prevent storage abuse.
- **Description/Rationale length**: cap at 2048 characters each at the tool layer.
- Flag: `needs-security-review` — escalate the `requestedBy` immutability gap: the CRD
  currently relies on RBAC (no update verb) rather than a webhook. A validating admission
  webhook could enforce immutability at the apiserver level. Recommend adding this in a
  follow-on security hardening issue.

## Resource & Performance Impact

- One new CRD (`proposeddatasources.iaf.io`); namespace-scoped.
- Each proposal is a small CR (< 2 KB). No controller reconciliation loop is required for
  the basic flow — operators approve via `kubectl patch status`.
- **AppURL population**: optionally, the application controller can write `appURL` to
  the status of any `ProposedDataSource` in the same namespace when its referenced app
  reaches Running. This requires the ApplicationReconciler to list ProposedDataSources
  in the same namespace during reconcile. Add this only if the operator workflow
  benefit justifies the added coupling.

## Migration / Compatibility

- No breaking changes to existing `DataSource` or `Application` CRDs.
- RBAC additions are additive.
- The data catalog resource JSON gains a `pendingProposals` field — agents that ignore
  unknown fields are unaffected.
